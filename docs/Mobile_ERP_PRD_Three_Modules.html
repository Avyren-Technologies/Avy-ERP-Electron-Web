HR Management, Machine Management & Visitor Management

A comprehensive functional specification for three core modules of the Mobile ERP application — a manufacturing solution for small and medium-sized enterprises.

Version 2.0 February 2026 Status: Draft Internal / Confidential HR Management Machine Management Visitor Management
Introduction & Scope

This document is a focused module-level PRD extracted from the Mobile ERP v2.0 full product specification. It covers three modules in complete detail: HR Management, Machine Maintenance, and Visitor Management. Each module is described end-to-end — functional requirements, data flows, user interactions, business rules, integration dependencies, and access controls.

ℹ️
The Mobile ERP is a mobile-first application serving manufacturing SMEs. All screens are designed for phone/tablet operation. Role-based access is enforced across every module — each user sees only the data and actions their role permits.

Module 09 · HR Management
HR Management
Employee records, attendance tracking, leave management, payroll processing, and production-based incentive configuration.

The HR Management module is the people-operations hub of the Mobile ERP. It handles the complete employee lifecycle — from onboarding and daily attendance through leave approval, salary computation, and incentive payouts. Attendance data is sourced from the Security module's gate-scan records, ensuring a single source of truth. Incentive calculations are driven by production output logged in the Production module.

9.2 Attendance

The Attendance section provides HR personnel with a daily view of workforce presence. Data is not manually entered here — it is sourced directly from the Security module's gate attendance system, which captures entry/exit via manual code entry or face scanning.

Daily Attendance Dashboard
Present — Number of employees who have checked in today and meet the working hour threshold
Absent — Employees with no gate record or below the minimum threshold
On Leave — Employees with approved leave for the selected date
Date selector to review historical attendance records
Business Rule
Working Hours Configuration

HR administrators configure minimum working hour thresholds used by the system to auto-categorise daily attendance for payroll purposes:

Full Day Threshold — e.g., ≥ 8 hours → categorised as Full Day
Half Day Threshold — e.g., ≥ 4 hours and < 8 hours → categorised as Half Day
Absent Threshold — Below the half-day minimum → categorised as Absent

These thresholds are configurable and feed directly into payroll deduction calculations for the month.

🔗
Data Source: All attendance records originate in the Security module (Section 5.1). Gate punch-in/out timestamps are automatically synced to HR. HR does not duplicate entry — it consumes and categorises the Security data.
9.3 Leave Management

Manages employee leave requests through an approval workflow. HR personnel review pending requests and take action, with the system notifying employees of the outcome.

Reject — Marks request rejected; notifies employee with rejection
9.4 Payroll

The Payroll section automates monthly salary calculations based on each employee's pre-configured salary structure, attendance records (full/half/absent days), and any applicable incentives from production output.

Monthly Payroll Summary
Total Employees on payroll for the selected month
Gross Salary — Sum of all earnings before deductions across all employees
Total Deductions — PF, ESI, tax, loans aggregated
Net Payable — Final disbursement amount
Per-employee rows showing individual gross, deductions, and net pay
Functional Requirement
Payroll Master Configuration (Per Employee)

Each employee has an individually configured salary structure defined in the Payroll Master. This structure drives the automated monthly payroll calculation.

Earnings Components
Basic Salary — Fixed base component
HRA (House Rent Allowance)
Conveyance Allowance
Special Allowance
Other Allowances — configurable additional components
Deduction Components
PF (Provident Fund) — Statutory deduction
ESI (Employee State Insurance) — Statutory deduction
Professional Tax / Income Tax (TDS)
Loan EMI — Advance/loan repayment deduction
Bank Details
Bank name, account number, IFSC code — used for payroll disbursement reference
⚙️
Calculation Logic: Monthly net pay = (Basic + All Allowances + Production Incentive) − (PF + ESI + TDS + Loan EMI) × (Attendance Factor based on working-day thresholds)
9.5 Incentive Configuration

Configures production-based incentive rules that are applied to employee earnings. Incentive payouts are automatically calculated using production quantities logged in the Production module against the slab rules defined here.

Functional Requirement
Production-Based Incentive Rule Setup

Each incentive rule is configured with the following parameters:

Part Selection (Multiple) — One or more finished goods / parts to which this rule applies
Operation Selection (Multiple) — One or more manufacturing operations (e.g., turning, milling, grinding) covered by this rule
Machine Selection (Multiple) — One or more machines on which qualifying production occurs
Target Production Quantity — The baseline output a worker must exceed before incentive slabs begin applying
Range-Based Slab Rules — Tiered payout rates per piece above the target
Example Slab Structure
Production RangeRate per PieceNotes
0 – 100 pcs₹ 0Below target — no incentive
101 – 120 pcs₹ 5 / pcSlab 1 — entry incentive
121 – 150 pcs₹ 8 / pcSlab 2 — mid-tier incentive
151+ pcs₹ 12 / pcSlab 3 — top performer
🔗
Automatic Calculation: Incentive amounts are auto-computed by the system using production slip data from the Production module. HR configures the rules; the Production module provides the quantities; the system does the math and adds the payout to that employee's monthly payroll.

Module 08 · Machine Maintenance
Machine Management
Preventive maintenance scheduling, breakdown tracking, spare parts inventory, and OEE monitoring for manufacturing equipment.

The Machine Management module ensures maximum equipment uptime through proactive preventive maintenance (PM) scheduling, rapid breakdown response, and spare parts inventory management. It also serves as the foundational source of machine data consumed by the Production module's OEE (Overall Equipment Effectiveness) dashboard.

Module Sub-Sections
SectionFeatureDescription
8.1Machine MasterMachine registration and PM schedule configuration per machine
8.2Maintenance ScheduleList and calendar view of due and upcoming PM tasks with execution workflow
8.3Breakdown ManagementReport, track, and resolve machine breakdowns with downtime logging
8.4Spare Parts InventoryStock of spare parts with low-stock alerts and machine linkage
8.5OEE IntegrationMachine data feeds the Production OEE dashboard
8.1 Machine Master

The Machine Master is the central registry of all manufacturing equipment. Every machine must be registered here before it can appear in maintenance schedules, production slips, or OEE reporting.

Functional Requirement
Machine Registration

Each machine record captures:

Machine Code — Unique identifier for the machine (e.g., CNC-001)
Machine Name — Descriptive name (e.g., Horizontal CNC Lathe)
Location — Shop floor location or bay designation
Manufacturer — OEM name
Model — Machine model number/name
Technical Specifications — Capacity, power rating, and other relevant specs
Functional Requirement
Preventive Maintenance (PM) Schedule Configuration

Each machine has one or more PM schedules configured directly within its master record:

Frequency — Daily, Weekly, Monthly, Quarterly, or Annually
Tasks Per Interval — Checklist of maintenance tasks to perform (e.g., "Lubricate spindle bearings", "Check coolant level")
Estimated Duration — Time required to complete the maintenance tasks
Required Spare Parts — Parts that need to be on hand for this PM (linked to Spare Parts Inventory)

Once configured, the system auto-generates scheduled tasks in the Maintenance Schedule section at the appropriate intervals.

📌
Master Integration: Machine records from this master are also referenced in the Masters Module (Section 11.3) and the Production module, ensuring a single source of truth for machine data across the application.
8.2 Maintenance Schedule

The operational heart of the maintenance module. Displays all due, upcoming, and historical preventive maintenance tasks and provides the workflow to execute them.

Maintenance Execution Workflow

When a maintenance technician opens a scheduled task and begins work:

Log Replacement Parts — Select from Spare Parts Inventory; stock automatically decremented

Completed tasks are archived and visible in maintenance history per machine.

8.3 Breakdown Management

Enables rapid reporting and tracking of unplanned machine stoppages. Breakdowns are the primary negative contributor to OEE Availability and must be tracked with precision for both resolution speed and root-cause analysis.

Functional Requirement
Breakdown Report (New Breakdown)

When a machine breaks down, the operator or maintenance staff raises a breakdown report containing:

Machine — Select from Machine Master
Issue Type — Category of failure (Mechanical, Electrical, Hydraulic, Software, Other)
Description — Detailed description of the symptom or failure
Priority — Critical High Medium Low
Reported Time — Auto-stamped at submission
Active Breakdowns List
Real-time list of all open/in-progress breakdowns across the facility
Downtime Tracking — Live elapsed downtime counter from time of report
Assigned Team — Maintenance team or technician responsible for resolution
Priority-sorted so critical breakdowns appear at the top
Breakdown Detail View
Machine name, location, issue description, issue type
Reported time, total downtime duration (running counter if unresolved)
Assigned maintenance team and technician
Resolution notes — entered when closing the breakdown
Parts used during repair — linked to spare parts inventory
Close Breakdown action — marks resolved, records resolution timestamp
📊
OEE Impact: Breakdown duration contributes directly to the Availability component of OEE for the affected machine. Closing a breakdown with accurate timestamps enables the Production module to compute correct Availability percentages.
8.4 Spare Parts Inventory

A dedicated inventory for machine spare parts, separate from the main goods inventory. Enables maintenance teams to verify part availability before scheduling maintenance and to replenish stock proactively.

SKU / Part Code — Unique identifier for each spare part
Part Name & Description — What the part is and its specifications
Applicable Machines — Which machines this part is compatible with
Stock Quantity — Current quantity on hand
Reorder Level — Minimum quantity threshold before alert is triggered
Low-Stock Alerts — Critical when below critical level, Low when approaching reorder level
🔗
Integration: When a maintenance task is executed (PM or breakdown repair) and parts are consumed, this inventory is automatically decremented. Parts required for PM tasks are pre-linked in the Machine Master PM configuration.
8.5 OEE Integration (Production Module)

Machine data registered in this module feeds directly into the Production module's OEE (Overall Equipment Effectiveness) dashboard. OEE is the primary equipment performance metric for manufacturing operations.

Reference: Production Module OEE Dashboard
OEE = Availability × Performance × Quality
Availability — (Planned time − Breakdown downtime) ÷ Planned time. Driven by Machine Master planned time and Breakdown Management downtime records.
Performance — Actual production rate ÷ Ideal production rate. Driven by Production slip entries.
Quality — Good units ÷ Total units (after scrap/NC deduction). Driven by Scrap & NC entries.
OEE Colour Indicators
≥ 85% World-class
60 – 84% Needs improvement
< 60% Critical

Module V · Visitor Management
Visitor Management
Pre-registration, gate check-in/check-out, QR-based self-registration, host notification, and visitor history tracking.

The Visitor Management module controls and records the entry of all external visitors to the facility. It supports advance visitor pre-registration by employees, walk-in visitor handling by security at the gate, QR-code-based self-registration, and a complete audit trail of all visits. Security personnel operate the gate-side verification, while employees (hosts) initiate pre-registrations and receive arrival notifications.

Module Sub-Sections
SectionFeatureDescription

V.2Pre-RegistrationAdvance registration of visitors by host employees before they arrive
V.3Visit HistorySearchable archive of all past visits with full details and timestamps
V.4QR Self-RegistrationWalk-in visitors scan a QR code and self-register via a web form
V.5Security IntegrationGate-side verification, host notification, and access approval
V.1 Today's Visitors

The primary operational view for the Visitor Management module. Provides a real-time snapshot of all visitor activity for the current day, split between expected (pre-registered) and walk-in visitors.

V.2 Pre-Registration

Host employees pre-register visitors before they arrive at the facility. Pre-registration streamlines gate processing and enables security to be aware of expected visitors in advance.

Functional Requirement
Visitor Pre-Registration Form

An employee (the host) fills in the following fields to register an upcoming visitor:

Visitor Name — Full name of the expected visitor
Company / Organisation — Visitor's employer or affiliation
Phone Number — Visitor's contact number
Purpose of Visit — Meeting, Delivery, Audit, Vendor, Interview, etc.
Person to Meet (Host) — The registering employee is auto-populated as host; can be changed to register on behalf of a colleague
Expected Date & Time — When the visitor is expected to arrive

On submission, the visitor appears in Today's Visitors (if scheduled for today) and in the Security module's Visitor Verification screen as an expected visitor.

🔔
Notification Flow: When Security checks in a pre-registered visitor at the gate, an automatic notification is sent to the host employee informing them their visitor has arrived.
V.3 Visit History

A complete, searchable audit trail of all visitor activity — past and present. Provides accountability, security audit capability, and statistical reporting.

Searchable by visitor name, company, host name, or date range
Each record shows: Visitor name, company, purpose, host, check-in time, check-out time, visit duration
Tapping any record shows full visit detail including any security notes
Filterable by visit type: Pre-registered, Walk-in, QR Self-Registered
Export capability for compliance and audit reporting
V.4 QR Code–Based Visitor Self-Registration

Designed to reduce security desk congestion for walk-in visitors. A QR code is displayed at the reception or printed and posted near the gate. Visitors scan it with their phone and complete a web form independently.

Functional Requirement
QR Self-Registration Flow
Step 1 — QR Code Display
Security generates and displays a persistent QR code (on a screen or printed sign) at the facility entrance
The QR code links to a mobile-optimised web registration form (no app required for the visitor)
Step 2 — Visitor Self-Registration Web Form
Name — Visitor's full name
Company / Organisation
Phone Number
Purpose of Visit
Person to Meet — Employee name / department
Step 3 — Host Approval
On form submission, the named host employee receives an in-app notification
Host can Approve or Decline the visit request
Step 4 — Security Notification
On host approval, security receives an alert to allow the visitor access
Visitor appears in the Security module's Visitor Verification screen as an approved walk-in
📱
No App Required: The visitor's self-registration form is a standard web page — visitors do not need to download or install the Mobile ERP app. The entire visitor interaction happens through their phone's browser after scanning the QR code.
V.5 Security Module Integration (Gate Verification)

The Security module's Visitor Verification screen is the gate-side complement to the Visitor Management module. Security personnel use it to check in visitors against pre-registrations and handle walk-ins.

Functional Requirement
Visitor Gate Management (Security Section 5.3)

The Security Visitor Verification screen shows today's expected visitors (sourced from Visitor Management pre-registrations). Security can perform the following actions:

View Full Details — Name, company, purpose, and host employee for each expected visitor
Verify & Check In — Marks visitor as Checked In; timestamps arrival; triggers host notification
Check Out — Marks visitor as Checked Out; timestamps departure; records total visit duration
Handle Walk-In Visitors — Manual entry for unannounced visitors not in the pre-registration list
Walk-in Handling via Security
Security can manually capture walk-in visitor details at the gate: name, company, phone, purpose, person to meet
Host employee is notified and can approve remotely
Walk-in visitors appear in the Visit History with type = "Walk-In"
